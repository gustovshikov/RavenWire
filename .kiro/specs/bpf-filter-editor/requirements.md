# Requirements Document: BPF Filter Editor

## Introduction

The RavenWire sensor stack uses Berkeley Packet Filter (BPF) expressions to control which network traffic is captured by the AF_PACKET ring buffer. A misconfigured BPF filter can either overload a sensor by capturing too much traffic (elephant flows flooding the pipeline) or blind it by excluding critical traffic. Today the Config Manager has no UI for managing BPF filter profiles — the only mechanism is manually editing the `config/sensor/bpf_filters.conf` file.

This feature adds a pool-level BPF Filter Editor to the Config Manager web UI: structured rule management for elephant flow exclusions, CIDR pair exclusions, and port exclusions; a raw BPF expression editor for advanced users; server-side BPF syntax validation and compilation; BPF profile versioning; a compiled expression preview; and integration with the existing RBAC, audit logging, pool management, and health monitoring systems.

BPF filter profiles are pool-scoped — all sensors in a pool share the same BPF profile. Saving a BPF profile does NOT automatically push it to sensors; deployment remains an explicit operator action through the existing deployment workflow. The sensor detail page already displays a `bpf_restart_pending` indicator per capture consumer; this feature connects the BPF editor to that existing health signal.

## Glossary

- **Config_Manager**: The Phoenix/LiveView web application that manages the RavenWire sensor fleet.
- **Sensor_Pool**: A named grouping of Sensor_Pods that share a common configuration profile, including BPF filter configuration. Stored in the `sensor_pools` table.
- **Sensor_Pod**: An individual sensor node enrolled in the Config_Manager. Each Sensor_Pod has an optional `pool_id` foreign key referencing a Sensor_Pool.
- **BPF_Profile**: A versioned BPF filter configuration associated with a Sensor_Pool. A BPF_Profile contains zero or more structured Filter_Rules and an optional Raw_BPF_Expression. The BPF_Profile is the unit of configuration that gets compiled, validated, and eventually deployed to sensors.
- **Filter_Rule**: A single structured exclusion entry within a BPF_Profile. Each Filter_Rule has a rule type (elephant flow, CIDR pair, or port exclusion), type-specific parameters, an enabled/disabled state, and an optional human-readable label. Filter_Rules are combined to generate the final BPF expression.
- **Elephant_Flow_Exclusion**: A Filter_Rule type that excludes high-bandwidth flows from capture based on a combination of source/destination CIDR, port, and protocol. Used to skip known bulk-transfer traffic such as storage replication, backup streams, and CDN traffic that would overwhelm the capture pipeline.
- **CIDR_Pair_Exclusion**: A Filter_Rule type that excludes traffic between a specific source CIDR and destination CIDR pair. Used to skip known trusted internal traffic paths such as management network communication or inter-cluster replication.
- **Port_Exclusion**: A Filter_Rule type that excludes traffic on specific ports or port ranges, optionally scoped to a protocol (TCP, UDP, or any). Used to skip known high-volume services that do not require packet capture.
- **Raw_BPF_Expression**: A free-form BPF filter string written in tcpdump/libpcap syntax. Advanced users can provide a raw expression that either replaces or supplements the structured Filter_Rules. The Raw_BPF_Expression is validated by the BPF_Compiler before saving.
- **BPF_Compiler**: A server-side component that validates and compiles BPF expressions using a libpcap-based compiler, such as a native `pcap_compile` wrapper or `tcpdump -d` invoked without a shell. The BPF_Compiler checks syntax correctness and produces compiled bytecode output for preview.
- **Compiled_Expression**: The final BPF filter string generated by combining all enabled Filter_Rules (and optionally the Raw_BPF_Expression) into a single expression. This is the expression that would be written to `bpf_filters.conf` on deployment.
- **BPF_Profile_Version**: A monotonically increasing integer on the BPF_Profile that increments each time the profile's filter configuration changes. Used for tracking configuration drift and deployment history.
- **BPF_Profile_Version_Record**: An immutable snapshot of a BPF_Profile at a specific BPF_Profile_Version, including the Compiled_Expression, Raw_BPF_Expression, composition mode, rule snapshot, actor, and timestamp.
- **Last_Deployed_BPF_Version**: The most recent BPF_Profile_Version explicitly deployed to the Sensor_Pool's sensors. Used to determine whether saved BPF changes are pending deployment.
- **RBAC_Gate**: The runtime permission check from the auth-rbac-audit spec that enforces role-based access on routes and LiveView events.
- **Audit_Entry**: An append-only record in the `audit_log` table capturing who performed what action, when, on which target, and whether it succeeded.
- **SensorAgentClient**: The existing Elixir module that dispatches control commands to Sensor_Agent instances via mTLS HTTP.
- **Health_Registry**: The existing GenServer that tracks sensor health state, including the `bpf_restart_pending` flag per capture consumer.

## Requirements

### Requirement 1: BPF Editor Page

**User Story:** As a sensor operator, I want a BPF filter editor page for each pool, so that I can view and manage the pool's BPF filter profile from a single location.

#### Acceptance Criteria

1. THE Config_Manager SHALL expose a BPF editor page at `/pools/:id/bpf` accessible to all authenticated Users with the `sensors:view` Permission.
2. THE Config_Manager SHALL display the pool's current BPF_Profile on the editor page, including: the list of structured Filter_Rules with their types, parameters, enabled states, and labels; the Raw_BPF_Expression if one exists; the Compiled_Expression preview; and the BPF_Profile_Version with last-updated metadata (timestamp and actor).
3. WHEN no BPF_Profile exists for a pool, THE Config_Manager SHALL display an empty state message indicating no BPF filter profile has been configured and prompting the User to create one.
4. THE Config_Manager SHALL display a "BPF Restart Pending" summary on the editor page showing the count of Sensor_Pods in the pool that currently have the `bpf_restart_pending` flag set in the Health_Registry, linking to the sensor detail page for each affected sensor.
5. THE Config_Manager SHALL display a notice on the editor page stating that saving BPF configuration does not automatically deploy it to sensors.
6. THE Config_Manager SHALL subscribe to PubSub updates for the pool so that changes made by other Users are reflected on the editor page without a full page reload.

### Requirement 2: Structured Filter Rule Management

**User Story:** As a sensor operator, I want to add, edit, remove, and reorder structured filter rules, so that I can build BPF exclusion profiles without writing raw BPF syntax.

#### Acceptance Criteria

1. THE Config_Manager SHALL support three Filter_Rule types: Elephant_Flow_Exclusion, CIDR_Pair_Exclusion, and Port_Exclusion.
2. WHEN a User adds an Elephant_Flow_Exclusion rule, THE Config_Manager SHALL accept the following parameters: source CIDR (optional), destination CIDR (optional), port or port range (optional), protocol (TCP, UDP, or any; default any), and a human-readable label (optional, max 255 characters). THE Config_Manager SHALL require at least one of source CIDR, destination CIDR, or port to be specified.
3. WHEN a User adds a CIDR_Pair_Exclusion rule, THE Config_Manager SHALL accept the following parameters: source CIDR (required), destination CIDR (required), and a human-readable label (optional, max 255 characters).
4. WHEN a User adds a Port_Exclusion rule, THE Config_Manager SHALL accept the following parameters: port or port range (required, single port or hyphen-separated range such as "1024-65535"), protocol (TCP, UDP, or any; default any), and a human-readable label (optional, max 255 characters).
5. THE Config_Manager SHALL validate CIDR values as valid IPv4 or IPv6 CIDR notation (e.g., `10.0.0.0/8`, `192.168.1.0/24`, `2001:db8::/32`).
6. THE Config_Manager SHALL validate port values as integers between 1 and 65535, and port ranges as two valid port values where the start port is less than or equal to the end port.
7. THE Config_Manager SHALL allow Users to enable or disable individual Filter_Rules without removing them from the profile. Disabled rules are excluded from the Compiled_Expression.
8. THE Config_Manager SHALL allow Users to reorder Filter_Rules within the profile. Rule order determines the order of clauses in the Compiled_Expression.
9. THE Config_Manager SHALL allow Users to edit existing Filter_Rules in place and to delete Filter_Rules from the profile.
10. THE Config_Manager SHALL display Filter_Rules in a list or table showing: rule type icon or label, a human-readable summary of the rule parameters, the enabled/disabled state, and edit/delete controls.
11. WHEN a User modifies Filter_Rules (add, edit, delete, reorder, or toggle), THE Config_Manager SHALL regenerate and display the Compiled_Expression preview in real time before the User saves.

### Requirement 3: Raw BPF Expression Editor

**User Story:** As an advanced sensor operator, I want to write or paste a raw BPF expression, so that I can create filters that go beyond what the structured rule builder supports.

#### Acceptance Criteria

1. THE Config_Manager SHALL provide a raw BPF expression text area on the editor page where Users can enter a free-form BPF filter string in tcpdump/libpcap syntax.
2. THE Config_Manager SHALL support two composition modes for the Raw_BPF_Expression: "append" mode where the raw expression is combined with structured Filter_Rules using a logical AND, and "replace" mode where the raw expression replaces all structured Filter_Rules entirely when the raw expression is non-empty.
3. WHEN the composition mode is "replace," THE Config_Manager SHALL visually indicate that structured Filter_Rules are inactive and will not be included in the Compiled_Expression, while preserving the rules in storage for potential reactivation.
4. WHEN the composition mode is "append," THE Config_Manager SHALL combine the structured Filter_Rules and the Raw_BPF_Expression into a single Compiled_Expression using logical AND, with the structured rules first and the raw expression appended in parentheses.
5. THE Config_Manager SHALL validate the Raw_BPF_Expression using the BPF_Compiler before allowing the User to save the profile.
6. WHEN the Raw_BPF_Expression is empty and the composition mode is "append," THE Config_Manager SHALL generate the Compiled_Expression from structured Filter_Rules only.
7. WHEN the Raw_BPF_Expression is empty and the composition mode is "replace," THE Config_Manager SHALL treat the Compiled_Expression as empty, display the no-filter warning, and allow saving only after the User confirms that all traffic will be captured.

### Requirement 4: BPF Compilation and Validation

**User Story:** As a sensor operator, I want the Config Manager to validate and compile BPF expressions before saving, so that invalid filters are caught before they can be deployed to sensors.

#### Acceptance Criteria

1. THE Config_Manager SHALL provide a server-side BPF_Compiler that validates BPF expressions by invoking a libpcap-based compiler on the Config_Manager server, either through a native wrapper or through `tcpdump -d`/`tcpdump -ddd` executed without a shell.
2. WHEN a User clicks "Validate" or "Save," THE Config_Manager SHALL compile the Compiled_Expression using the BPF_Compiler and display the result: success with the compiled bytecode instruction count, or failure with the compiler error message.
3. IF the BPF_Compiler reports a syntax error in the Compiled_Expression, THEN THE Config_Manager SHALL reject the save operation, display the compiler error message to the User, and highlight the portion of the expression that caused the error when identifiable.
4. THE Config_Manager SHALL execute BPF compilation asynchronously so that the LiveView remains responsive during compilation. THE Config_Manager SHALL display a loading indicator while compilation is in progress.
5. THE Config_Manager SHALL bound BPF compilation with a configurable timeout, defaulting to 5 seconds, to prevent runaway compiler processes from blocking the server.
6. IF the BPF_Compiler process times out, THEN THE Config_Manager SHALL report a timeout error to the User and reject the save operation.
7. THE Config_Manager SHALL NOT pass BPF expressions through a system shell. IF an external command is used, THEN THE Config_Manager SHALL invoke it with parameterized command execution (for example, `System.cmd/3` with an argument list) or another shell-free API, and SHALL reject expressions that cannot be safely passed to that API.
8. THE Config_Manager SHALL provide a "Validate" button separate from "Save" so that Users can check expression validity without committing changes.
9. WHEN the Compiled_Expression is valid, THE Config_Manager SHALL display the compiled BPF bytecode instruction count as a complexity indicator to help operators assess filter overhead.
10. WHEN the Compiled_Expression is empty because no filter is configured, THE Config_Manager SHALL skip external BPF compilation, treat the expression as valid, and clearly indicate that no capture filter will be applied.

### Requirement 5: BPF Expression Generation

**User Story:** As a sensor operator, I want the structured rules automatically compiled into a valid BPF expression, so that I can see exactly what filter will be applied to the capture pipeline.

#### Acceptance Criteria

1. THE Config_Manager SHALL generate a Compiled_Expression from enabled structured Filter_Rules by combining each rule's BPF clause with logical AND; each structured exclusion SHALL produce a `not (...)` clause, and all enabled clauses SHALL be joined in rule order.
2. THE Config_Manager SHALL generate BPF clauses for each Filter_Rule type as follows:
   - **Elephant_Flow_Exclusion**: `not (` + predicates joined with `and` + `)`, where predicates may include `src net <src_cidr>`, `dst net <dst_cidr>`, `port <port>` or `portrange <start>-<end>`, and `tcp` or `udp`.
   - **CIDR_Pair_Exclusion**: `not (src net <src_cidr> and dst net <dst_cidr>)`.
   - **Port_Exclusion**: `not (` + predicates joined with `and` + `)`, where predicates include `port <port>` or `portrange <start>-<end>` and optionally `tcp` or `udp`.
3. THE Config_Manager SHALL display the Compiled_Expression in a read-only preview area with syntax highlighting or monospace formatting, updated in real time as Filter_Rules are modified.
4. WHEN no enabled Filter_Rules exist and no Raw_BPF_Expression is set, THE Config_Manager SHALL display an empty Compiled_Expression with a message indicating that all traffic will be captured (no filter applied).
5. THE Config_Manager SHALL generate syntactically valid tcpdump/libpcap BPF expressions. FOR ALL valid combinations of enabled Filter_Rules, generating the expression and then compiling it with the BPF_Compiler SHALL succeed (round-trip property), except for an empty expression, which is treated as the intentional no-filter state.

### Requirement 6: BPF Profile Versioning

**User Story:** As a sensor operator, I want BPF profiles versioned so that I can track what changed, when, and by whom, and correlate deployed versions with sensor health.

#### Acceptance Criteria

1. THE Config_Manager SHALL maintain a BPF_Profile_Version integer on each BPF_Profile that starts at 1 when the profile is first created.
2. WHEN a User saves changes to a BPF_Profile and the submitted configuration differs from the stored configuration, THE Config_Manager SHALL increment the BPF_Profile_Version by exactly 1 and update the last-modified timestamp and actor.
3. THE Config_Manager SHALL display the current BPF_Profile_Version, last-modified timestamp, and last-modified actor on the editor page.
4. THE Config_Manager SHALL NOT increment the BPF_Profile_Version when a validation-only operation is performed (the "Validate" button).
5. THE Config_Manager SHALL create a BPF_Profile_Version_Record for each created or updated version so that the exact filter and rule snapshot at any saved version can be retrieved.

### Requirement 7: BPF Profile CRUD

**User Story:** As a sensor operator, I want to create, view, and reset BPF profiles for pools, so that I can manage the capture filter lifecycle.

#### Acceptance Criteria

1. WHEN a User navigates to `/pools/:id/bpf` for a pool that has no BPF_Profile, THE Config_Manager SHALL display an option to create a new empty profile.
2. WHEN a User creates a new BPF_Profile, THE Config_Manager SHALL initialize it with BPF_Profile_Version 1, no Filter_Rules, no Raw_BPF_Expression, composition mode "append," an empty Compiled_Expression, and record an Audit_Entry with action `bpf_profile_created`.
3. WHEN a User saves changes to an existing BPF_Profile, THE Config_Manager SHALL update the profile atomically (all rule changes, raw expression, composition mode, Compiled_Expression, and version snapshot in one transaction), increment the version only when the submitted configuration differs from the stored configuration, and record an Audit_Entry with action `bpf_profile_updated` containing the old and new version numbers and a summary of changes.
4. THE Config_Manager SHALL provide a "Reset to Empty" action that clears all Filter_Rules and the Raw_BPF_Expression, sets composition mode to "append," stores an empty Compiled_Expression, increments the version, and records an Audit_Entry with action `bpf_profile_reset`. THE Config_Manager SHALL require confirmation before executing the reset.
5. WHEN a Sensor_Pool is deleted, THE Config_Manager SHALL cascade-delete the associated BPF_Profile and all its Filter_Rules, consistent with the pool deletion behavior defined in the sensor-pool-management spec.

### Requirement 8: RBAC Integration

**User Story:** As a platform admin, I want BPF filter management actions protected by role-based access control, so that only authorized users can modify capture filters that affect sensor visibility.

#### Acceptance Criteria

1. THE Config_Manager SHALL use the canonical `bpf:manage` Permission from the auth-rbac-audit spec for BPF write operations (create profile, save profile, reset profile, add/edit/delete/reorder/toggle rules, edit raw expression, validate expression).
2. THE Config_Manager SHALL verify that the auth-rbac-audit Policy grants `bpf:manage` to the `sensor-operator`, `rule-manager`, and `platform-admin` Roles, consistent with the `pools:manage` and `forwarding:manage` Permission grant patterns.
3. THE Config_Manager SHALL grant read-only BPF access (`sensors:view`) to all authenticated Roles, allowing viewing of the BPF editor page, Filter_Rules, Raw_BPF_Expression, Compiled_Expression, and version metadata.
4. WHEN a User without the `bpf:manage` Permission attempts a BPF write action via LiveView event, THE RBAC_Gate SHALL deny the action, display an error flash, and record an Audit_Entry with action `permission_denied`.
5. THE Config_Manager SHALL hide BPF write UI elements (Add Rule button, Save button, Reset button, Validate button, rule edit/delete/toggle/reorder controls, raw expression text area) from Users whose Role does not include the `bpf:manage` Permission.
6. THE Config_Manager SHALL enforce RBAC on every LiveView `handle_event` callback for BPF write actions, regardless of whether the UI element is hidden.

### Requirement 9: Audit Logging for BPF Actions

**User Story:** As an auditor, I want all BPF filter management actions recorded in the audit log, so that changes to the capture filter — which directly affects sensor visibility — are traceable and attributable.

#### Acceptance Criteria

1. THE Config_Manager SHALL record an Audit_Entry for each of the following BPF actions: `bpf_profile_created`, `bpf_profile_updated`, `bpf_profile_reset`, `bpf_expression_validated`, and `bpf_rule_toggled`.
2. EACH BPF-related Audit_Entry SHALL contain: the actor identity (username or API_Token name), the actor type, the action name, `target_type` set to `bpf_profile`, `target_id` set to the BPF_Profile's ID, the result (`success` or `failure`), and a JSON detail field with action-specific context.
3. THE `bpf_profile_updated` Audit_Entry detail SHALL include: the old and new BPF_Profile_Version numbers, a summary of rule changes (rules added, removed, modified, reordered), and whether the Raw_BPF_Expression or composition mode changed.
4. THE Config_Manager SHALL write BPF-related Audit_Entries for mutations within the same database transaction as the BPF mutation, so that if the audit write fails, the BPF mutation is rolled back.
5. THE Config_Manager SHALL include the Compiled_Expression text in the `bpf_profile_updated` and `bpf_profile_created` audit details so that the exact filter at each version is recorded in the audit trail.

### Requirement 10: No Automatic Deployment on Save

**User Story:** As a sensor operator, I want saving BPF configuration to be separate from deploying it, so that I can review and validate changes before they affect live capture.

#### Acceptance Criteria

1. THE Config_Manager SHALL NOT automatically push BPF configuration to sensors when a BPF_Profile is created, updated, or reset.
2. THE Config_Manager SHALL display a clear notice on the BPF editor page after every save operation stating that the configuration has been saved but not deployed, and that sensors will continue using their current BPF filter until an explicit deployment is performed.
3. WHEN a BPF_Profile has been saved with a BPF_Profile_Version greater than the Last_Deployed_BPF_Version for the pool, THE Config_Manager SHALL display a visual indicator such as "Saved changes pending deployment" on the BPF editor page and the pool detail page.

### Requirement 11: BPF Data Model

**User Story:** As an engineer implementing the BPF filter editor, I want a well-defined data model for BPF profiles and filter rules, so that the configuration is stored reliably and supports versioning.

#### Acceptance Criteria

1. THE Config_Manager SHALL create a `bpf_profiles` table via an Ecto migration with the following columns: `id` (binary_id primary key), `pool_id` (binary_id foreign key to `sensor_pools`, unique, on_delete cascade), `version` (integer, not null, default 1), `last_deployed_version` (integer, nullable), `raw_expression` (text, nullable), `composition_mode` (text, not null, default "append"), `compiled_expression` (text, nullable), `updated_at` (utc_datetime_usec), `updated_by` (text, nullable), `inserted_at` (utc_datetime_usec).
2. THE Config_Manager SHALL create a unique index on `pool_id` in the `bpf_profiles` table to enforce one BPF_Profile per Sensor_Pool.
3. THE Config_Manager SHALL create a `bpf_filter_rules` table via an Ecto migration with the following columns: `id` (binary_id primary key), `bpf_profile_id` (binary_id foreign key to `bpf_profiles`, on_delete cascade), `rule_type` (text, not null), `params` (map/jsonb, not null, type-specific parameters), `label` (text, nullable), `enabled` (boolean, not null, default true), `position` (integer, not null), `inserted_at` (utc_datetime_usec), `updated_at` (utc_datetime_usec).
4. THE Config_Manager SHALL create an index on `(bpf_profile_id, position)` in the `bpf_filter_rules` table to support ordered retrieval.
5. THE Config_Manager SHALL validate `rule_type` values against the allowed set: `elephant_flow`, `cidr_pair`, `port_exclusion`.
6. THE Config_Manager SHALL validate `composition_mode` values against the allowed set: `append`, `replace`.
7. THE Config_Manager SHALL cascade-delete BPF_Profiles and their Filter_Rules when the parent Sensor_Pool is deleted.
8. THE Config_Manager SHALL validate the `params` field against the expected schema for each `rule_type`:
   - **elephant_flow**: `{"src_cidr": string|null, "dst_cidr": string|null, "port": integer|null, "port_end": integer|null, "protocol": "tcp"|"udp"|"any"}` with at least one of `src_cidr`, `dst_cidr`, or `port` non-null.
   - **cidr_pair**: `{"src_cidr": string, "dst_cidr": string}` with both required.
   - **port_exclusion**: `{"port": integer, "port_end": integer|null, "protocol": "tcp"|"udp"|"any"}` with `port` required.
9. THE Config_Manager SHALL create a `bpf_profile_versions` table via an Ecto migration with the following columns: `id` (binary_id primary key), `bpf_profile_id` (binary_id foreign key to `bpf_profiles`, on_delete cascade), `version` (integer, not null), `raw_expression` (text, nullable), `composition_mode` (text, not null), `compiled_expression` (text, nullable), `rules_snapshot` (map/jsonb, not null), `created_by` (text, nullable), and `inserted_at` (utc_datetime_usec).
10. THE Config_Manager SHALL create a unique index on `(bpf_profile_id, version)` in the `bpf_profile_versions` table to prevent duplicate version snapshots.
11. THE Config_Manager SHALL update `last_deployed_version` only when an explicit deployment workflow successfully deploys that BPF_Profile_Version to the pool.

### Requirement 12: BPF Restart Pending Integration

**User Story:** As a sensor operator, I want the BPF editor to show which sensors have a BPF restart pending, so that I can correlate filter changes with sensor state and know when a restart is needed.

#### Acceptance Criteria

1. THE Config_Manager SHALL query the Health_Registry for `bpf_restart_pending` status of all Sensor_Pods in the pool and display the count of sensors with pending BPF restarts on the BPF editor page.
2. WHEN one or more Sensor_Pods in the pool have `bpf_restart_pending` set to true, THE Config_Manager SHALL display a warning banner on the BPF editor page listing the affected sensor names with links to their sensor detail pages.
3. THE Config_Manager SHALL subscribe to PubSub messages for `bpf_restart_pending` state changes so that the warning banner updates in real time without page reload.
4. WHEN no Sensor_Pods in the pool have `bpf_restart_pending` set, THE Config_Manager SHALL display a green status indicator confirming all sensors are running with their current BPF filter.

### Requirement 13: Pool Navigation Integration

**User Story:** As a sensor operator, I want the BPF editor accessible from the pool detail page navigation, so that I can find it alongside other pool configuration pages.

#### Acceptance Criteria

1. THE Config_Manager SHALL add a "BPF Filters" navigation link to the pool detail page tab bar, alongside existing links for Config, Sensors, Forwarding, and Deployments.
2. THE Config_Manager SHALL display a BPF summary on the pool detail/show page including: whether a BPF_Profile exists, the current BPF_Profile_Version, the number of enabled Filter_Rules, and whether a Raw_BPF_Expression is configured.
3. WHEN the pool's BPF_Profile has a BPF_Profile_Version greater than the Last_Deployed_BPF_Version, THE Config_Manager SHALL display a "pending deployment" badge on the BPF navigation link.
